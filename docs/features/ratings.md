# Ratings

## Purpose

Этот документ задаёт бизнес-правила для core ratings.

## Rules

- используются две фиксированные оценки: `overcomplication`, `importance`;
- допустимые значения: целые числа `0..5` или `null`;
- оценки принадлежат конкретному work item;
- leaf-узлы показывают собственные оценки;
- parent-узлы показывают агрегированные read values;
- редактирование оценок не должно менять структуру дерева.
- для primary tree view каноническим источником мгновенного обновления агрегатов является client-derived state, а не обязательный повторный `fetch` после каждого `PATCH`;
- успешное изменение leaf-оценки должно в одном пользовательском цикле обновить и сам leaf, и все затронутые parent aggregate values;
- `PATCH /api/work-items/[id]` может оставаться row-scoped и не обязан возвращать агрегаты предков, если клиент после подтверждённого сохранения локально пересчитывает суммы для затронутой ветки;
- full tree reload остаётся только reconciliation-механизмом для сверки с серверной истиной, а не штатным способом показать корректные суммы после обычного редактирования leaf rating.

## Client Contract

- локальный пересчёт aggregate sums обязателен для всех подтверждённых leaf rating edits независимо от того, были ли они инициированы кликом по rating control, blur-сохранением, undo/redo или другой канонической data-action;
- пересчёт должен затрагивать только изменённый leaf и его ancestor chain, без обязательного полного refresh дерева;
- parent rows остаются read-only для ratings и не получают собственные editable rating values;
- один и тот же leaf edit flow не должен иногда обновлять parent sums локально, а иногда ждать forced refetch.

## Negative Scenarios

- если `PATCH` завершился ошибкой, клиент не должен показывать новые aggregate sums как подтверждённый результат;
- если локальная история или queued save приводит к повторному применению устаревшего ответа, stale response не должен откатывать более новые aggregate sums;
- если после refresh серверное дерево расходится с локально пересчитанным состоянием, клиент обязан принять серверную версию как источник истины и пересобрать локальную проекцию;
- если узел перестал быть leaf до применения rating patch, операция должна завершиться canonical error path и не менять агрегаты частично.
