# Workspace Metrics

## Purpose

Этот документ фиксирует feature-specific правила для workspace-scoped метрик, которые заменяют legacy `blocksMoney`.

## Rules

- набор метрик принадлежит конкретному workspace;
- у разных workspace может быть разный набор метрик, включая пустой набор;
- каждая метрика содержит обязательное поле `shortName` и необязательное поле `description`;
- оба текстовых поля при сохранении обрезают пробелы по краям;
- пустой `shortName` запрещён;
- дубликаты `shortName` и `description` внутри одного workspace допустимы;
- максимальная длина имени и описания в первой версии отдельно не ограничивается приложением;
- новая метрика добавляется в конец списка метрик workspace;
- ручное изменение порядка метрик в первой версии не поддерживается;
- удаление метрики удаляет и саму метрику, и все значения этой метрики у всех work items данного workspace;
- удаление метрики не требует отдельного подтверждения в UI;
- удаление метрики обязано быть частью канонической session undo/redo системы;
- legacy `blocksMoney` удаляется из модели и не сохраняется как совместимый alias.

## Value Model

- значение метрики у work item использует один canonical enum: `none`, `indirect`, `direct`;
- `none` является значением по умолчанию;
- пустое значение или отсутствие записи интерпретируется как `none`;
- leaf-узлы показывают и редактируют собственные значения метрик;
- parent-узлы не хранят и не редактируют собственные значения метрик в primary tree view;
- parent-узлы показывают только агрегированное значение по потомкам;
- агрегирование выполняется по приоритету `direct > indirect > none`;
- если среди потомков есть хотя бы одно `direct`, parent показывает `direct`;
- иначе, если среди потомков есть хотя бы одно `indirect`, parent показывает `indirect`;
- иначе parent показывает `none`;
- после подтверждённого изменения значения метрики клиент обязан локально пересчитать агрегаты по ancestor chain без обязательного полного refresh дерева.

## UI Rules

- управление метриками вынесено в popup настроек конкретного workspace;
- popup настроек является единственной канонической точкой для rename workspace, add/edit/delete метрик и удаления workspace;
- inline-управление метриками через заголовки таблицы, hover `+` у колонок и double click по имени колонки не входит в первую версию;
- в дереве каждая метрика отображается отдельным столбцом;
- leaf-ячейка метрики в первой версии редактируется через dropdown;
- parent-ячейка метрики остаётся read-only;
- короткое имя метрики используется как заголовок столбца;
- описание метрики отображается только там, где это уместно в настройках workspace или вспомогательном UI, и не обязано всегда занимать место в таблице;
- визуальный стиль popup и столбцов должен наследовать уже существующий интерфейс: спокойная плотная компоновка, минимальный шум, типографическая иерархия и без новых декоративных паттернов.

## Negative Scenarios

- если создание метрики завершилось ошибкой, popup должен сохранить ввод пользователя и не создавать частично сохранённую колонку;
- если rename метрики завершился ошибкой, старое подтверждённое имя должно остаться источником истины, а неуспешный ввод должен остаться recoverable;
- если удаление метрики завершилось ошибкой, ни сама метрика, ни её значения не должны исчезнуть частично;
- если dropdown edit значения метрики завершился ошибкой, клиент не должен закреплять новый агрегат как подтверждённый результат;
- если локальный пересчёт агрегатов расходится с серверной версией после reconcile, клиент обязан принять серверную версию как источник истины;
- если undo удаления метрики не может быть применён атомарно, система не должна оставлять полувосстановленную метрику или частично возвращённые значения.
